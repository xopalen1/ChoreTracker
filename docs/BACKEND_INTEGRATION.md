# Backend Integration Guide

This frontend expects a REST API. Adjust the routes if needed, but keep responses consistent.

Chores and chat are backend-authoritative in the current frontend setup.

## Base URL

Current frontend assumes same-origin routes:

- `/api/chores`
- `/api/chores/:id`

If your backend is on another origin, you can update fetch URLs in `js/main.js`.

## Endpoints

### 1) List chores

- Method: `GET`
- Route: `/api/chores`
- Response: `200 OK`

```json
[
  {
    "id": "abc123",
    "title": "Take out trash",
    "assignee": "Alex",
    "assignedDate": "2026-04-09",
    "dueDate": "2026-04-10",
    "isDone": false,
    "isDeleted": false,
    "dueDateHistory": [],
    "status": "open"
  }
]
```

### 2) Create chore

- Method: `POST`
- Route: `/api/chores`
- Body:

```json
{
  "title": "Clean bathroom sink",
  "assignee": "Sam",
  "dueDate": "2026-04-11"
}
```

- Response: `201 Created` (or `200 OK`)
- Response body: created chore with `id`
- Backend rule: `assignedDate` is generated by backend and set to Sunday.

### 3) Update chore status

- Method: `PATCH`
- Route: `/api/chores/:id`
- Body:

```json
{
  "isDone": true
}
```

- Response: updated chore object

### 4) Update due date (with history)

- Method: `PATCH`
- Route: `/api/chores/:id`
- Body:

```json
{
  "dueDate": "2026-04-15",
  "dueDateHistory": [
    {
      "previousDueDate": "2026-04-11",
      "newDueDate": "2026-04-15",
      "editedAt": "2026-04-09T18:03:10.000Z"
    }
  ]
}
```

- Response: updated chore object

### 5) Delete chore

- Method: `PATCH`
- Route: `/api/chores/:id`
- Body:

```json
{
  "isDeleted": true
}
```

- Response: updated chore object

### 5.1) Check chore state

- Method: `GET`
- Route: `/api/chores/:id`
- Use response booleans for quick checks:
  - `isDone`
  - `isDeleted`

### 6) List chat messages

- Method: `GET`
- Route: `/api/messages`
- Response: `200 OK`

```json
[
  {
    "id": "m1",
    "text": "Please remember to wipe counters after cooking",
    "sentAt": "2026-04-09T18:12:24.000Z"
  }
]
```

### 7) Create chat message

- Method: `POST`
- Route: `/api/messages`
- Body:

```json
{
  "text": "Can someone start laundry tonight?"
}
```

- Response: `201 Created` (or `200 OK`)
- Response body:

```json
{
  "id": "m2",
  "text": "Can someone start laundry tonight?",
  "sentAt": "2026-04-09T18:13:02.000Z"
}
```

## Validation Rules Recommended

- `title`: required, string, max 80 chars
- `assignee`: required, string, max 40 chars
- `assignedDate`: recommended ISO date string (YYYY-MM-DD) set by backend
- `dueDate`: required, ISO date string (YYYY-MM-DD)
- `isDone`: required boolean
- `isDeleted`: required boolean
- `dueDateHistory`: required array (can be empty)
- `dueDateHistory[].previousDueDate`: ISO date string
- `dueDateHistory[].newDueDate`: ISO date string
- `dueDateHistory[].editedAt`: ISO datetime string
- Chat `text`: required string, max 240 chars
- Chat `sentAt`: ISO datetime string

## C23 Backend Note

Because you are building the backend in pure C23, keep your JSON shape strict and predictable:

- Use fixed keys exactly as shown in this document.
- Treat `assignedDate` and `dueDate` as plain `YYYY-MM-DD` strings.
- Return consistent response schemas for all endpoints.
- Storing chores in a CSV database is fine as long as API responses keep the same JSON shape.


## CORS

If frontend and backend run on different ports/origins, enable CORS on your API.

## Auth

No auth is implemented on frontend. Add your preferred auth and update request headers where needed.
